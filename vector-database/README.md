## What is a Vector Database?

A **vector database** is a specialized database designed to **store**, **index**, and **search** high-dimensional vector representations of data known as **embeddings**.

Unlike traditional databases that rely on exact matches (e.g. SQL `WHERE` clauses), vector databases use **similarity search** techniques — such as **cosine similarity** or **Euclidean distance** — to find items that are semantically or visually similar.

| Traditional Database | Vector Database |
|---------------------|-----------------|
| Stores structured rows and columns | Stores dense vector representations |
| Exact match queries | Approximate nearest neighbor (ANN) search |
| B-tree, hash indexes | HNSW, IVF, LSH, flat indexes |
| Keyword / predicate filtering | Semantic / similarity-based retrieval |

---

## Embeddings

**Embeddings** are dense numerical representations of data — words, sentences, images, or audio — mapped into a continuous high-dimensional space where similar items are positioned closer together.

- Generated by machine learning models that capture semantic meaning, context, and relationships within the data
- Enable similarity measurement through distance metrics instead of direct comparison of raw text or media
- Support semantic search, recommendation systems, clustering, classification, and cross-lingual matching

### Use Cases Enabled by Embeddings

| Domain | Description |
|--------|-------------|
| Semantic search | Retrieve documents by meaning, not lexical overlap |
| Recommendation systems | Surface items with similar embedding proximity |
| Clustering | Group data points by vector neighborhood |
| Classification | Assign labels based on embedding space boundaries |
| Cross-lingual matching | Align representations across languages in a shared space |

## How Embeddings Work

Embeddings convert raw data (text, images, audio) into dense numerical vectors that preserve meaning and structural relationships.

```
Input (text / image / audio)
        │
        ▼
Feature extraction model
(Transformer, CNN, etc.)
        │
        ▼
Fixed-length vector in ℝⁿ
        │
        ▼
Similar items → proximal vectors
Dissimilar items → distant vectors
```

1. **Input processing** — raw data is passed through a trained model (e.g. transformer for text, CNN for images)
2. **Feature extraction** — the model encodes salient features of the input
3. **Vector encoding** — features are projected into a fixed-length vector in high-dimensional space
4. **Spatial arrangement** — semantically similar inputs cluster; dissimilar inputs diverge
5. **Similarity computation** — proximity in vector space is measured via distance or similarity metrics

This enables downstream tasks — search, recommendations, classification — to operate on **semantic proximity** rather than exact string or pixel-level matching.

## Similarity Search

At query time, the pipeline follows:

```
Query → Embedding model → Query vector
Query vector → ANN index → Top-K nearest neighbors
```

### Distance Metrics

| Metric | Formula Concept | Typical Use |
|--------|-----------------|-------------|
| **Cosine similarity** | Angle between vectors; invariant to magnitude | Text embeddings, normalized vectors |
| **Euclidean distance (L2)** | Straight-line distance in vector space | General-purpose similarity |
| **Dot product** | Inner product of vectors | When magnitude carries signal |

Vector databases implement these metrics with indexing structures (HNSW, IVF, PQ) to avoid brute-force comparison across the full corpus at scale.

## Implementation (FAISS)

Stores 3 sample vectors and performs similarity search using **L2 distance**. The `query_vector` is compared against all stored vectors; indices and distances of the top-2 nearest neighbors are returned.

```python
import faiss
import numpy as np

data_vectors = np.array([
    [0.1, 0.2, 0.3, 0.4],
    [0.2, 0.1, 0.4, 0.3],
    [0.9, 0.8, 0.7, 0.6],
], dtype='float32')

dimension = data_vectors.shape[1]
index = faiss.IndexFlatL2(dimension)
index.add(data_vectors)

query_vector = np.array([[0.1, 0.2, 0.3, 0.35]], dtype='float32')
distances, indices = index.search(query_vector, k=2)

print("Indices of closest vectors:", indices)
print("Distances from query:", distances)
```

**Output:**

```
Indices of closest vectors: [[0 1]]
Distances from query: [[0.0025 0.0325]]
```

Vectors at indices `0` and `1` are nearest to the query; index `2` is an outlier with substantially higher L2 distance.

## Applications

### Image and Video Search

Feature embeddings are extracted from images or video frames and indexed in the vector database. Query embeddings retrieve the most visually similar media via ANN search.

### Question Answering Systems

Queries and knowledge-base passages are embedded into a shared space. Vector similarity retrieves the most relevant context chunks, outperforming keyword-only retrieval for paraphrased or semantically equivalent queries.

### Cross-Lingual Information Retrieval

Multilingual embedding models project text from different languages into a shared vector space, enabling query in one language and retrieval of relevant documents in another.

### Fraud and Anomaly Detection

Embeddings of normal behavioral patterns are stored as a reference distribution. Deviation in embedding distance flags anomalous or potentially fraudulent activity.

## Role in RAG

In a Retrieval-Augmented Generation pipeline, the vector database is the retrieval layer:

```
Documents → Chunk → Embed → Index in Vector DB
                                    │
User Query → Embed → ANN Search → Top-K chunks
                                    │
                          Augment prompt → LLM → Response
```

| Phase | Function |
|-------|----------|
| **Indexing** | Persist chunk embeddings with optional metadata filters |
| **Retrieval** | Return top-K semantically relevant chunks for a query |
| **Update** | Re-embed and upsert when the knowledge base changes |

The quality of RAG output is directly bounded by retrieval recall and ranking accuracy at this layer.
